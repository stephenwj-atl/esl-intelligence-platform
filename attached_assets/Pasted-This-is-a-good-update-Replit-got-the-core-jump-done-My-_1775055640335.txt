What is still weak

	•	country differentiation is muted

That is the first thing to fix.

The second most important issue is Gap 5.

If PERS does not recalculate after data refresh, then you do not yet have a living intelligence platform. You have a scoring engine that runs once. For a real DFI-style system, updated source data must propagate into project risk.

The third important issue is Gap 6.

Priority order

	2.	Add automatic PERS recalculation on data refresh
	3.	Convert portfolio views to aggregate PERS, capital mode, and monitoring intensity

	6.	Replace reference-data pipelines with live endpoints where possible

One thing I would tighten conceptually

The brief says:

Project Overlay — Sector-specific complexity, adjusted by SEA and ESIA mitigation

That wording is a bit muddy.

Overlay should really mean location and project-context exposure.
Sector-specific complexity belongs more naturally in Sensitivity or Intervention Risk.

Not fatal, but later you may want to clean the definitions so each component is clearly distinct:
	•	CERI = country baseline
	•	Overlay = project/site/location modifiers
	•	Sensitivity = sector / receptor / governance / disaster-history susceptibility
	•	Intervention Risk = delivery-model risk

That makes the model easier to defend.

Bottom line

The build is real.

Gap 1 — Wire regional_data into live PERS calculation

Current issue

The PERS engine accepts governance quality and disaster loss history inputs, but project creation currently falls back to hardcoded defaults:
	•	governance = 50
	•	disaster history = 40

The regional_data table is already populated by:
	•	INFORM Risk Index
	•	Transparency International CPI
	•	EM-DAT

But the /api/projects flow is not querying and passing the values into calculatePERS().

Required fix

Update the project creation and project reassessment flow so that, at assessment time, the backend:
	1.	identifies the project country
	2.	queries regional_data for the latest values relevant to that country
	3.	extracts:
	•	CPI / governance indicator
	•	EM-DAT disaster loss / disaster history indicator
	•	INFORM composite / coping-vulnerability indicator if available
	4.	normalizes them if necessary
	5.	passes them into calculatePERS() as real parameters rather than defaults

Implementation requirements
	•	no hardcoded fallback unless no country data exists
	•	if fallback is used, log it and expose it in explainability
	•	persist the source values used in the assessment record
	•	store provenance for each sensitivity input:
	•	source name
	•	dataset key
	•	timestamp
	•	fallback status

Acceptance

A Barbados project and a Guyana project should no longer receive identical sensitivity sub-scores unless the underlying country values truly match.

Gap 5 — Automatic PERS recalculation when data refreshes

Current issue

When pipeline data changes, existing project PERS scores do not recalculate.

Required fix

Implement a recalculation mechanism.

Required behavior

When relevant source data updates, the system must:
	1.	identify which countries were affected
	2.	identify active projects in those countries
	3.	recalculate PERS for those projects
	4.	store both old and new values
	5.	write change records into risk_history or equivalent
	6.	avoid duplicate recalculations for the same ingestion batch

Architecture

Implement one of:
	•	post-ingestion hook
	•	job queue
	•	scheduled reconciliation worker

Prefer a robust pattern that supports future scaling.

Persist change tracking

For each recalculation store:
	•	prior PERS
	•	new PERS
	•	delta
	•	changed inputs
	•	triggering source(s)
	•	recalculation timestamp

Acceptance

If CPI or EM-DAT changes for a country, affected projects should receive updated assessments without manual project recreation.
