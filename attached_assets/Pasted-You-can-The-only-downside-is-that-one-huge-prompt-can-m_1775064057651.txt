# 8. IMPLEMENT INSTRUMENT-SPECIFIC MODIFIER RULES

Do not let instruments arbitrarily overwrite profiles. Use bounded and auditable modifiers.

## 8.1 Loan modifiers

Loans should:

* increase the effect of low confidence
* increase penalty for low implementation readiness
* increase penalty for low insurability or poor asset resilience where relevant
* support stricter decline logic

Use bounded equivalent adjustments such as:

* low confidence: +3 to +10
* low readiness: +2 to +8

## 8.2 Grant modifiers

Grants should:

* reduce automatic decline pressure from pure physical hazard where impact rationale is strong
* increase importance of outcome delivery risk
* increase importance of implementation capacity
* increase importance of milestone readiness

Use bounded adjustments such as:

* weak theory of change: +4 to +10
* weak implementation readiness: +4 to +10
* strong impact rationale with strong controls: partial offset allowed

## 8.3 Blended modifiers

Blended should:

* increase transition readiness importance
* increase milestone importance
* increase instrument structure quality importance
* support outputs like `GRANT_FIRST`, `BLENDED_NOW`, `LOAN_READY_AFTER_VALIDATION`

Bounded adjustments such as:

* missing transition pathway: +4 to +10
* weak concessional logic: +2 to +6
* strong milestone structure: -2 to -6
* strong validation plan: -2 to -5

## 8.4 TA / Programmatic / Emergency modifiers

Support lower exact-site dependency, but not weak monitoring logic.
Higher focus on:

* governance
* readiness
* implementation
* monitoring
* staged controls

Store modifiers as auditable rule sets.

# 9. IMPLEMENT DECISION MATRICES BY INSTRUMENT

Do not let one generic threshold control all instruments.

## 9.1 Loan decision matrix

Use logic like:

* low final score + adequate confidence + acceptable readiness => `PROCEED`
* moderate score or moderate confidence => `CONDITION`
* high score or very weak readiness/confidence => `DECLINE`

## 9.2 Grant decision matrix

Use logic like:

* low/moderate score + adequate delivery structure => `PROCEED`
* high score but high strategic rationale + adequate controls => `PROCEED_WITH_CONTROLS`
* weak baseline / weak design => `DEFER_PENDING_BASELINE`
* weak design but fixable => `RESEQUENCE`
* excessive risk without credible delivery path => `DO_NOT_FUND`

## 9.3 Blended decision matrix

Use logic like:

* ready for debt with grant support => `BLENDED_NOW`
* not debt-ready but de-riskable => `GRANT_FIRST`
* needs more validation => `CONDITION`
* no credible path to transition => `DO_NOT_PROCEED`

Store decision matrix metadata structurally.

# 10. IMPLEMENT CALIBRATION GUARDRAILS

Calibration must be constrained.

## 10.1 Top-level weight guardrails

* no top-level weight shifts more than ±0.10 from seeded family default without methodology review
* no top-level layer below 0.05 or above 0.50 without explicit review
* base layer weights must normalize to 1.00

## 10.2 Subfactor guardrails

* subfactor relevance stays between 0.00 and 1.00
* changes >0.20 from default require a calibration note
* if more than 5 subfactors materially change, status becomes `UNDER_REVIEW`

## 10.3 Confidence guardrails

* confidence adjustment may not exceed bounded equivalent, e.g. ±15 risk points without explicit approval
* formula must be documented and inspectable
